Building Portico

From Portico

Jump to: ,

The prospect of building Portico from scratch can be a little intimidating for those unfamiliar with the layout of the repository or the basics of Apache Ant. Every effort has been made to ensure that building Portico is as simple and straightforward as possible, after all, it is something we have to do all the time! As a result, depending on what exactly you are attempting to do, it is possible to trigger a full build with just a single, small command. This page describes the basic structure of the repository and the process required to build Portico.

[edit] Requirements

Portico bundles virtually everything you need to build it in the repository so that you don't have to manage the process of downloading and installing it yourself. There are only two exceptions this rule: compilers and installation tools.

The exact system requirements to build Portico depend what parts of it you are trying to build. The full list of possible requirements is outlined below, along with some information on what each dependency is for.

Note that although there are a number of requirements specified, most of them are only necessary under certain circumstances. The only mandatory requirement is the Java Development Kit (JDK).
Portico Building Requirements
Requirement Needed For
Java 7.0 or Greater (required) Everything. No escaping this one.
C++ Compiler (MSVC 8/9/10 or GCC) (generally required) If you want to build the C++ interface and/or the C++ unit tests. The following compilers are supported by Portico and the build system: 
# Visual Studio 2005 (VC8)
# Visual Studio 2008 (VC9)
# Visual Studio 2010 (VC10)
# GCC 4+ (Linux)
# GCC 4+ via the XCode command line tools (Mac OS X)
Nullsoft Scriptable Install System (optional) If you want to build windows installers. Install it and set the environment variable NSIS_HOME to point at its location.

Note that some tasks will have a dependency on others, and those other tasks may have requirements. For example to create a full release build the unit tests need to run (and they require CppUnit).


[edit] Building Portico

The Portico build process is powered by a highly customized Apache Ant setup with a number of customer-written tasks. The Portico repository contains its own pre-configured copy of Ant, along with helper scripts to invoke it.

The build process contains a number of build targets that control it. To run the build, you just use the supplied ant execution scripts and provide that target name. For example (where "sandbox" is the target name): 

[tim@zapp:codebase]$ ./ant sandbox

You can also chain targets together, so that multiple are executed. For example, to wipe the slate clean and create an entirely fresh sandbox (as is often useful after getting the latest code with an svn update), you would issue the following command: 

[tim@zapp:codebase]$ ./ant clean sandbox

The build process is split up into a number of high level tasks, each of which can be executed for either the Java or C++ interface builds, or both. The primary targets are described below:

Main Portico Build Targets
Command What it Does
./ant clean Remove all build artefacts. Use when you want to create a fresh build.
./ant sandbox This target creates a "sandbox" in which you can test your changes without having to generate an installer and install Portico. A sandbox looks just like the directory structure you would get if you ran an installer, except that it can be built on the fly. When the sandbox is generated, it is created in the codebase/dist/portico-version directory.

Use the sandbox when you want to test any changes (either changes you made, or changes that have come in as a result of a Git update) without going to the bother of generating an installer and installing a new version of Portico. This is quite handy if you don't have the installer tools installed, and thus CAN'T generate an installer. You can just generate the sandbox and point the RTI_HOME environment variable at it!

./ant release Everything. It will create a complete Portico distribution from scratch. This command will:
  1. Run a clean
  2. Compile all the code
  3. Run all the unit tests (lengthy)
  4. Generate the installers, placing them in the codebase/dist directory

Use it when you want to generate a complete, tested build. The unit tests will take a long time, and if any of them fail, the build will fail. If you just want a new Portico jar file containing small changes you made to the RTI quickly, then use the "sandbox".

./ant compile Compiles all the Portico code and puts it in the build directory
./ant test Compile and run all the unit tests (you will need CppUnit installed to run the C++ unit tests - if you don't have it, the build will fail)
./ant installers Runs all the installation targets to generate platform-specific installers. This wil generate the installers from the sandbox contents.
If you ever need help figuring out what the name of a target is or what is available on a particular platform, you can call ./ant -projecthelp and it will print a list of the major build targets for that platform along with brief descriptions of what each target does.

[edit] The Two Build Halves

The build process itself is made up of two separate halves: one for the Java code (the main RTI and the Java interfaces) and the other for the C++ interface. You can tell which build targets belong to which side by checking their prefix. Targets that begin with java. belong to the java build, while targets beginning with cpp. belong to the C++ interface build.

For example, consider the following two targets compile the java and c++ interface code respectively: 

[tim@zapp:codebase]$ ./ant java.compile (would compile the java code)
[tim@zapp:codebase]$ ./ant cpp.compile (would compile the c++ interface code)

There may be times when you are only interested in using part of the Portico build process. For example, you may have changed some of the main RTI code and want to run the unit tests, but seeing as you haven't changed any C++ code, you don't want to run those tests (alternatively, you might not have CppUnit installed, and thus can't run the C++ unit tests). In such circumstances, you would use one of the module-specific targets: 

[tim@zapp:codebase]$ ./ant java.test

For each of the primary build targets (see list above), there are java/c++ specific versions. These versions are prefixed with either java. or cpp. as appropriate.

Remember that if you are using one of these targets, they won't build anything related to the other module. For example, using ./ant java.sandbox won't cause the C++ interface libraries to be created and put in the sandbox. If they already reside there (and you didn't run a clean) then you should be fine. If you don't need these parts of Portico, there is no need to worry about this note

[edit] The C++ Build

Portico supports a number of compilers on Windows. It will also build both 32-bit and 64-bit versions of each library as well as debug and release flavours. The way the Portico build determines what it should attempt to compile is determined by the C++ "build profile". By default, it will attempt to build everything, but you can restrict this to less if you want.

When invoking the C++ interface build, you can specify values from three categories (compiler, architecture and type) to control what the Portico build will try and generate. By specifying this profile, Portico will determine which combinations to build. If you specify a value from a particular category, only those values specified will be built. If you don't specify any value from a category, all types for that category will be built.

For example, consider the following: 

./ant -Dprofile="vc8,vc10,debug" cpp.compile

To fill out the profile you just specify any of the supported options from any of the categories in a comm-separated list (order is unimportant). The command above will build all C++ interfaces using the VC8 and VC10 compilers (VC9 will be skipped). It will also only build the debug versions of the interfaces, but not the release versions. Notice that no specification about whether to build 32-bit or 64-bit libraries has been provided. This means that both will be generated.

The categories and their available options are listed below:

C++ Build Profile Categories
Category Options
Compiler (Windows) vc8, vc9, vc10 (Mac/Linux) gcc4
Architecture x86, amd64
Type debug, release

[edit] Generating Installers

You don't need to generate an installer if you don't want to. To use a version of Portico that you built yourself, just point the RTI_HOME environment variable at the codebase/dist/portico-version directory and run all of the executables from in there

As a convenience, Portico provides a number of installer packages for various platforms. Like everything else, the generation of these installers is also controlled by the build process. It is also important to note that the build process will only attempt to build an installer relevant to the platform it is currently running on. For example, you can only build a windows installer if you are running the build on windows.

[edit] Creating an Installer

There is a dedicated build target for creating installers. This target will go through and generate installers based on the contents of the sandbox: 

[tim@zapp:codebase]$ ./ant installers

The generated installers will be placed in the codebase/dist directory and will be given the name <coed>portico-[version]-[platform].[exe/zip/tar.gz]

[edit] When the Build Fails

Each part of the build process will have its own special requirements. If your system doesn't meet those requirements, you can't run that part of the build. For example, you can't compile the C++ interface without having a C++ compiler installed.

Obviously there is little you can do about this situation. If you don't have the required software, you can't build that part of Portico. If you happen not to care about that particular part of Portico, you could skip it by invoking a specific sequence of targets that doesn't include the piece you are missing.

[edit] I don't have a C++ Compiler

You can't build the C++ interface. However, you CAN build the rest of Portico. In this case, just use the java. prefixed targets (java.sandbox, java.release, java.test, ...)

[edit] I don't have all the C++ Compilers

You can restrict the scope of what the C++ side will attempt to build via the C++ build profile. See above.

[edit] Someone told me to "try the latest source"

Many people wish to benefit from the bug fixes and feature enhancements that go into the Portico codebase before a new official release is made. Obviously this is something the development team encourages, as it helps us get feedback and bug reports faster.

Alternatively, from time-to-time a developer may suggest that you "try the latest source" or the "latest trunk". To build a new version of Portico from the latest code requires only two commands:

  1. Get the code from Git
  2. Run an ./ant clean sandbox

This will generate a fully usable, complete Portico installation in the form of a sandbox (located in the codebase/dist/portico-version directory). If you want, you can just point your RTI_HOME environment variable at this directory and run happily as if you were using a version of Portico you got from an installer. This way, whenever you update to the latest code, you just have to run the sandbox target again.
